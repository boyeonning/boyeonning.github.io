## 1. CORS란?

**서로 다른 사이트 간에 데이터를 주고받을 때, 허용할지 말지를 결정하는 보안 정책**

### 비유: 아파트 경비원
> 당신이 **A 아파트** 주민인데, **B 아파트**에 방문하려고 하면
> B 아파트 경비원이 "당신 A 아파트 사람이죠? 여기 들어와도 됩니까?" 라고 확인하는 것!

---

## 2. Origin이란?

**Origin = 프로토콜 + 도메인 + 포트** 이 세 가지 조합

| 항목 | 예시 |
|------|------|
| 프로토콜 | `https://` |
| 도메인 | `naver.com` |
| 포트 | `:8080` |

셋 중 **하나라도 다르면** 다른 Origin으로 간주

```
https://naver.com  ≠  http://naver.com       ← 프로토콜 다름
https://naver.com  ≠  https://kakao.com      ← 도메인 다름
https://naver.com  ≠  https://naver.com:8080 ← 포트 다름
```

---

## 3. CORS 동작 방식

### 흐름 예시 (9000번 사이트 → 8000번 서버 요청)

```
① 사용자가 localhost:9000 접속
② 브라우저가 9000번 서버에 HTML 파일 요청
③ 9000번 서버가 HTML + JS 파일을 응답

④ JS 코드 안에 이런 코드가 있음:
   fetch('http://localhost:8000/orders')  ← 다른 서버(8000)에 요청!

⑤ 브라우저가 8000번 서버에 요청을 보낼 때
   Header에 "나 9000번에서 왔어~" 를 자동으로 포함

⑥ 8000번 서버가 응답할 때
   "9000번은 내가 허용한 곳이야!" 라고 응답에 포함
   Access-Control-Allow-Origin: http://localhost:9000

⑦ 브라우저가 이걸 확인하고 → JS에 데이터 전달 ✅
```

### 만약 허용 안 했다면?
```
8000번 서버가 9000번을 허용 목록에 안 넣었으면
→ 브라우저가 응답을 받아도 JS에 안 넘겨줌
→ CORS 에러 발생! ❌
```

> **핵심:** CORS 체크는 서버가 아니라 **브라우저**가 한다!
> 서버는 응답을 보내고, 브라우저가 판단해서 막거나 통과시킴.

---

## 4. CORS를 왜 쓰냐? (보안 때문!)

### CORS가 없다면?

```
1. 사용자가 bank.com 에 로그인
2. 실수로 hacker.com 도 열어버림
3. hacker.com의 JS 코드가 내 브라우저에서 실행되면서
   → 몰래 bank.com 에 송금 요청을 보냄!
4. 브라우저엔 이미 bank.com 로그인 쿠키가 있으니
   → 인증도 통과해버림 💸
```

### CORS가 있다면?

```
hacker.com 의 JS가 bank.com 에 요청을 보내려 하면
브라우저: "hacker.com은 bank.com이 허용한 Origin 아닌데?"
→ 차단! ❌
```

## 5. JS 코드가 자동으로 요청한다는 게 무슨 말?

### 내가 네이버에 접속할 때 실제로 일어나는 일

```
내가 naver.com 주소창에 입력
→ 네이버 서버가 HTML + JS 파일을 내 브라우저에 보내줌
→ 그 JS 코드가 내 컴퓨터에서 실행되면서
   → 알아서 뉴스 불러오고
   → 알아서 날씨 불러오고
   → 알아서 광고 불러오고...
```

> 내가 아무것도 안 했는데 페이지가 채워지는 것 = JS가 알아서 요청 보낸 것!

### 사람이 직접 하는 행동 vs JS가 하는 행동

| 구분 | 예시 | CORS 체크 |
|------|------|-----------|
| 사람이 직접 | 주소창 입력, 링크 클릭 | ❌ 없음 |
| JS가 자동으로 | fetch(), axios() 등 | ✅ 있음 |

> CORS는 **"사람이 의도하지 않은, 코드가 몰래 보내는 요청"** 을 막는 것!

## 6. 허용 범위 설정

### 은행처럼 민감한 서비스
```
Access-Control-Allow-Origin: https://bank.com
```
→ 자기 자신만 허용! 외부 사이트는 모두 차단

### 공개 API처럼 누구나 써도 되는 서비스
```
Access-Control-Allow-Origin: *
```
→ 모든 사이트 허용

| 서비스 | 설정 | 이유 |
|--------|------|------|
| 은행, 쇼핑몰 | 자기 도메인만 허용 | 민감한 데이터 보호 |
| 공공 날씨 API | `*` (전체 허용) | 누구나 써도 됨 |
| 구글 폰트, CDN | `*` (전체 허용) | 모든 사이트에서 불러와야 함 |

## 7. Preflight Request (사전 확인 요청)

### 비유: 식당 예약 전화

> 식당에 바로 가는 게 아니라,
> **"거기 자리 있어요? 저 채식인데 괜찮아요?"** 먼저 전화로 물어보는 것!

### 언제 Preflight가 발생하냐?

브라우저가 "이건 좀 민감한 요청인데?" 싶을 때 자동으로 먼저 확인

| 조건 | 예시 |
|------|------|
| 특수 메소드 사용 | `PUT`, `DELETE`, `OPTIONS` |
| JSON 데이터 전송 | `Content-Type: application/json` |
| 인증 헤더 포함 | `Authorization: Bearer xxx` |

> GET/POST로 단순 텍스트 보내는 건 Preflight 없이 바로 요청!

### Preflight 흐름

```
일반 요청:    [요청] → [응답]                       (2단계)

Preflight:   [사전확인] → [허락] → [요청] → [응답]  (4단계)
```

**① Preflight Request** — 브라우저가 먼저 물어봄
```
"야 서버야, 나 이 Origin인데 이 메소드로 요청해도 돼?"
```

**② Preflight Response** — 서버가 허용 목록 알려줌
```
✅ Origin: 허용
✅ Methods: GET, PUT, POST 허용
✅ Headers: Content-Type 허용
```

**③ 실제 Request** — 허락 받았으니 진짜 요청 전송

**④ 실제 Response** — 데이터 수신 🎉

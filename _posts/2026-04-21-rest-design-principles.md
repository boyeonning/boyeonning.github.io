## 1. 균일한 인터페이스 (Uniform Interface)

**핵심 아이디어:** 어떤 클라이언트든 **똑같은 방식**으로 서버와 소통한다.

### 비유
카페를 생각해보세요. 손님이 누구든 상관없이 앱으로 주문하든, 직접 말하든, 키오스크를 쓰든 **주문 방식이 통일**되어 있습니다.

REST도 마찬가지입니다. 모바일 앱이든, 웹 브라우저든, 다른 서버든 **같은 URL, 같은 HTTP 메서드**로 요청합니다.

### 실제 예시

`/users/1` 이라는 리소스가 있다면:

| 동작 | 메서드 | URL |
|------|--------|-----|
| 조회 | `GET` | `/users/1` |
| 수정 | `PUT` | `/users/1` |
| 삭제 | `DELETE` | `/users/1` |

누가 요청하든, 어떤 기기에서 오든 **이 규칙은 항상 동일**합니다.

### 장점
- **예측 가능** → API 문서만 보면 바로 사용 가능
- **독립성** → 클라이언트와 서버를 따로 개발 가능
- **일관성** → 배우기 쉽고 실수가 줄어듦

> **한 줄 요약:** "리소스마다 주소가 하나, 행동은 HTTP 메서드로만" 이라는 약속

## 2. 클라이언트-서버 독립성 (Client-Server)

**핵심 아이디어:** 클라이언트와 서버는 **서로 몰라도 된다.**

### 비유
TV 리모컨을 생각해보세요.
- 리모컨(클라이언트)은 TV 내부 회로를 몰라도 됩니다
- TV(서버)는 리모컨이 어떻게 생겼는지 몰라도 됩니다
- 둘 사이의 약속은 단 하나 → **"채널 올려"** 신호만 맞으면 OK

### 실제 예시

```
[클라이언트]                    [서버]
웹 브라우저   ──GET /users/1──▶  사용자 정보 반환
모바일 앱     ──GET /users/1──▶  (똑같이 반환)
다른 서버     ──GET /users/1──▶  (똑같이 반환)
```

- 서버는 클라이언트가 **React인지 Vue인지 iOS인지** 신경 안 씁니다
- 클라이언트는 서버가 **Java인지 Python인지 DB가 뭔지** 신경 안 씁니다

### 장점

| 상황 | 결과 |
|------|------|
| 앱 디자인을 바꾸고 싶다 | 서버 건드릴 필요 없음 |
| 서버 언어를 Java → Python으로 바꾼다 | 앱 수정 없음 |
| 앱을 웹 / 모바일 / 데스크탑 동시 출시 | 서버 하나로 모두 대응 |

> **한 줄 요약:** "약속된 API 형식만 지키면, 서로 내부 사정은 몰라도 된다"

## 3. 무상태 (Stateless)

**핵심 아이디어:** 서버는 클라이언트를 **기억하지 않는다.**

### 비유
**기억상실 웨이터** — 매번 테이블에 올 때마다 처음 보는 손님처럼 대합니다.

그래서 손님이 매번 주문할 때 **모든 정보를 직접 말해야** 합니다.
> "저 3번 테이블, 아메리카노 한 잔, 카드 결제요" ✅

### 실제 예시

**❌ 상태 있는 방식 (Stateful)**
```
요청1: "로그인" → 서버가 "이 사람 로그인했다" 기억
요청2: "내 정보 줘" → 서버가 기억을 꺼내서 응답
```

**✅ 무상태 방식 (Stateless)**
```
요청1: "로그인" → 토큰 발급
요청2: "내 정보 줘 + 토큰: abc123" → 매번 토큰을 직접 들고 옴
```

### 장점

| 장점 | 이유 |
|------|------|
| 서버 확장이 쉬움 | 어떤 서버가 받아도 처리 가능 |
| 장애에 강함 | 서버 하나 죽어도 다른 서버가 이어받음 |
| 단순함 | 서버가 상태 관리를 안 해도 됨 |

> **한 줄 요약:** "요청 한 번에 필요한 정보를 모두 담아 보내라, 서버는 아무것도 기억하지 않는다"

## 4. 캐시 가능 (Cacheable)

**핵심 아이디어:** 자주 쓰는 응답은 **저장해뒀다가 재사용**한다.

### 비유
**카페 메뉴판** — 손님이 올 때마다 주방에 물어보는 대신, 메뉴판을 벽에 붙여두면 됩니다. 메뉴가 바뀔 때만 메뉴판을 새로 교체하면 됩니다.

### 실제 동작 방식

```
첫 번째 요청
클라이언트 ──GET /products──▶ 서버
클라이언트 ◀──응답 + "1시간 캐시해도 됨"── 서버

두 번째 요청 (1시간 이내)
클라이언트 → 서버 안 물어보고 저장된 응답 바로 사용 ✅
```

### 캐시 가능 vs 불가능

| 데이터 | 캐시 가능? | 이유 |
|--------|-----------|------|
| 상품 목록 | ✅ | 자주 안 바뀜 |
| 공지사항 | ✅ | 자주 안 바뀜 |
| 내 장바구니 | ❌ | 사람마다 다름 |
| 실시간 주식 가격 | ❌ | 매초 바뀜 |

### 장점

| 장점 | 내용 |
|------|------|
| 빠름 | 서버까지 안 가도 됨 |
| 서버 부담 감소 | 같은 요청 반복 처리 안 해도 됨 |
| 네트워크 절약 | 데이터를 덜 주고받음 |

> **한 줄 요약:** "변하지 않는 데이터는 저장해두고 재사용해서, 속도는 올리고 서버 부담은 줄인다"

## 5. 계층화된 시스템 아키텍처 (Layered System)

**핵심 아이디어:** 클라이언트는 **자신이 누구와 직접 통신하는지 몰라도 된다.**

### 비유
**택배 배송** — 내가 쿠팡에서 주문하면 → 물류센터 → 배송대행 → 기사님 → 나.
나는 중간 과정을 전혀 몰라도 됩니다.

### 실제 구조

```
클라이언트
    ↓
[로드 밸런서]   ← 요청을 여러 서버에 분산
    ↓
[보안 서버]     ← 인증/해킹 차단
    ↓
[API 서버]      ← 실제 처리
    ↓
[데이터베이스]
```

클라이언트는 로드 밸런서에만 요청을 보내고, **뒤에 뭐가 있는지 전혀 알 필요 없습니다.**

### 장점

| 장점 | 내용 |
|------|------|
| 보안 | 클라이언트가 DB에 직접 접근 불가 |
| 유연성 | 중간 계층을 마음대로 추가/교체 가능 |
| 확장성 | 트래픽 많으면 서버만 늘리면 됨 |

> **한 줄 요약:** "중간에 뭐가 있든 클라이언트는 모른다. 각 계층은 자기 역할만 한다"

## 6. 온디맨드 코드 (Code on Demand)

**핵심 아이디어:** 서버가 클라이언트에게 **실행 가능한 코드를 보내줄 수 있다.**

### 비유
**앱 업데이트 없이 기능 추가**하는 느낌입니다.
- 일반적인 방식: 새 기능 → 앱 업데이트 → 사용자가 다운로드
- 온디맨드 코드: 서버가 코드를 직접 내려보냄 → 바로 실행 ✅

### 실제 예시

```html
<!-- 서버가 이런 코드를 클라이언트에 전송 -->
<script src="https://api.example.com/widget.js"></script>
<!-- 클라이언트는 받은 코드를 그냥 실행 -->
```

우리가 매일 쓰는 **웹사이트가 바로 이 방식**입니다. 브라우저가 서버에서 HTML/CSS/JS를 받아서 실행하죠.

### 실생활 예시

| 서비스 | 온디맨드 코드 |
|--------|-------------|
| 구글 지도 | 지도 위젯 JS를 서버에서 받아 실행 |
| 카카오 로그인 버튼 | 버튼 코드를 서버에서 내려받음 |
| 웹사이트 전체 | HTML/JS/CSS 모두 서버에서 받음 |

> ⚠️ REST에서 **유일하게 선택사항**인 원칙입니다. 보안 문제가 생길 수 있어서 꼭 써야 하는 건 아닙니다.

> **한 줄 요약:** "필요할 때 서버에서 코드를 받아 실행한다. 웹의 JS가 대표적인 예"

## 전체 요약

| 원칙 | 핵심 한 줄 |
|------|-----------|
| 균일한 인터페이스 | 리소스마다 주소 하나, 행동은 HTTP 메서드로만 |
| 클라이언트-서버 독립 | API 형식만 지키면 서로 내부 사정 몰라도 됨 |
| 무상태 | 요청마다 필요한 정보를 모두 담아 보내라 |
| 캐시 가능 | 안 바뀌는 데이터는 저장해서 재사용 |
| 계층화된 아키텍처 | 중간에 뭐가 있든 클라이언트는 모른다 |
| 온디맨드 코드 | 필요할 때 서버에서 코드를 받아 실행 (선택사항) |
